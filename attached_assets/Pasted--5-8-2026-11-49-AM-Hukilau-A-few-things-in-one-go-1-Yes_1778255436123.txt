3. About GitHub vs local changes — how to avoid trampling each other

You’re right to be concerned: with Replit pushing a lot of changes, we need a clean contract so server-side fixes don’t get clobbered.

Here’s the safe pattern:

• a) Treat GitHub main as the single source of truth for code.

  • Any code changes that matter long-term should live in the repo, not only on the server.
  • When Replit’s agent makes a fix, it pushes to github.com/jaysails67/sailhatteras_site on main.
  • On the server we deploy by:
    • git fetch && git reset --hard origin/main
    • Rebuild (e.g. pnpm --filter @workspace/api-server build)
    • Restart PM2
That’s what we did with the image URL fix: we forcibly synced the server to origin/main.

• b) Keep “server-only” configuration out of Git.
Things that should not be overwritten by Replit:

  • .env for secrets and server-specific values:
    • PORT=3001
    • BASE_PATH=/
    • DATABASE_URL=postgresql://…
    • SESSION_SECRET=…
    • Stripe / SMTP / Telegram keys, etc.
  • Apache vhost includes:
    • Those userdata/ssl/2_4/…/sailhatteras_spa.conf files are part of the server config, not the app repo.
These live outside of the repo or are in .gitignore, so Git pulls from Replit won’t touch them.

• c) Avoid making “one-off” code edits directly on the server.
